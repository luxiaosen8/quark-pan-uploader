# 上传并发、UI 一致性与性能优化 Clarify

## 目的

本文件用于收敛当前 spec 与 implementation plan 中仍未最终拍板、但会直接影响实现路径和验收口径的关键问题。每一项都包含：

- 需要澄清的问题
- 推荐答案
- 推荐原因
- 如果采用相反选择的影响

## 已确认上下文

- 当前项目为 Python 3.12 + PySide6 Windows 桌面工具。
- 上传链路目前是 `controller -> worker -> executor -> uploader`。
- `UploadWorker.run()` 当前按 job 串行。
- `UploadExecutionEngine.execute_job()` 当前按文件串行。
- `QuarkFileUploader._upload_multiple_parts()` 当前按 chunk 串行。
- `AppSettings` 当前尚未包含并发或 UI 节流相关设置项。
- `OfficialLoginDialog` 当前使用 `threading.Thread` 校验 Cookie。

## 澄清项 1：默认任务并发数与分片并发数应是多少？

### 问题

默认情况下，`job_concurrency` 和 `part_concurrency` 应该设置为多少，才能在提升吞吐的同时避免把远端接口、磁盘读取和 UI 压垮？

### 推荐答案

- 默认 `job_concurrency = 2`
- 默认 `part_concurrency = 3`
- 安全上限建议：
  - `job_concurrency <= 4`
  - `part_concurrency <= 6`

### 推荐原因

1. 当前项目从完全串行起步，直接把默认值拉高会增加远端限流和本地资源争抢风险。
2. `job_concurrency = 2` 足够让“批量子文件夹上传”场景获得第一波可见收益，同时便于定位问题。
3. `part_concurrency = 3` 对大文件分片上传通常已经能明显优于串行，又不会让鉴权请求和并行 PUT 数量过快膨胀。
4. 后续仍可通过 benchmark 再调优默认值，但第一版实现更应该偏稳妥。

### 若采用更激进方案

- 好处：某些网络条件下吞吐可能更高。
- 风险：更容易触发远端限流、日志刷屏、UI 卡顿和错误放大，首版稳定性更差。

## 澄清项 2：并发配置是否需要暴露到 UI？

### 问题

任务并发数、分片并发数、UI 刷新节流参数是否需要直接展示给用户在界面上调节？

### 推荐答案

第一阶段**不暴露到主界面**，仅保留在本地设置模型与配置文件中。

### 推荐原因

1. 当前用户主流程已经包含 Cookie、来源选择、目标目录选择和任务启动，继续增加并发调参会抬高操作复杂度。
2. 并发参数属于偏工程化设置，首版更适合先在 `AppSettings` 中稳定落地，再根据用户反馈决定是否上浮到 UI。
3. 先隐藏配置能缩小 UI 改动面，让本次目标聚焦在“能力提升 + 交互统一 + 性能稳定”。

### 若改为 UI 公开配置

- 好处：高级用户可自行调优。
- 风险：需要额外设计输入校验、默认值说明、错误提示和帮助文案，也会扩大测试面。

## 澄清项 3：登录弹窗是否必须切到 Qt 线程模型？

### 问题

`OfficialLoginDialog` 现在用 `threading.Thread` 做 Cookie 校验。是必须统一切换到 Qt 线程模型，还是只统一状态反馈与样式？

### 推荐答案

第一阶段建议：**优先统一状态反馈与信号回调方式；线程模型统一列为高优先级次级目标，但不是首个硬门槛。**

### 推荐原因

1. 当前主矛盾是上传并发与 UI 交互一致性，登录弹窗线程模型不是当前最大瓶颈。
2. 如果直接在同一轮同时推进上传并发改造 + 登录线程模型切换，风险叠加过高。
3. 先统一状态语义、样式契约和信号出口，后续再将底层执行切换到 Qt 线程模型，迁移成本更低。

### 具体落地建议

- 必做：统一 busy / validating / success / waiting 状态反馈。
- 必做：保证 UI 更新仍通过安全回调返回主线程。
- 可延后：把 `threading.Thread` 替换为 `QThread` / Qt 异步执行。

### 若立即强制统一到 Qt 线程模型

- 好处：线程模型更一致，后续维护更整齐。
- 风险：需要同时改动登录流程生命周期、回调与测试桩，超出本轮最小必要范围。

## 澄清项 4：性能验收使用什么基线样本？

### 问题

若没有固定样本集，“并发比串行快”会变成不可复现的主观判断。需要怎样的样本集才能稳定比较？

### 推荐答案

使用三组固定基线：

1. **小文件集**：200~500 个小文件，覆盖批量任务场景
2. **大文件集**：2~5 个大文件，覆盖 multipart 并发场景
3. **混合集**：少量大文件 + 多量小文件，覆盖真实使用场景

### 推荐原因

1. 小文件集主要验证 `job_concurrency` 的收益。
2. 大文件集主要验证 `part_concurrency` 的收益。
3. 混合集能暴露 UI 节流、日志聚合和资源边界问题。

### 最低记录指标

- 总耗时
- 成功任务数 / 失败任务数
- 平均吞吐
- 是否出现明显 UI 卡顿
- 是否出现远端限流或鉴权异常

## 澄清项 5：结果写入是否需要线程安全保护？

### 问题

当前 `UploadExecutionEngine` 通过 `result_writer.append_event()` 和 `append_share_result()` 写结果。引入 job 并发后，是否要做显式串行保护？

### 推荐答案

**需要。** 第一版实现应默认把 result writer 视为非线程安全对象，并在并发路径增加最小化保护。

### 推荐原因

1. 当前代码没有体现 result writer 的并发安全保证。
2. 即使底层文件写入“碰巧可用”，并发事件顺序混乱也会影响诊断和结果可读性。
3. 这里做最小化串行保护，比事后排查日志乱序或 JSON 竞争写入成本低得多。

### 推荐实现方向

- 方案 A：worker / controller 聚合后主线程串行写
- 方案 B：result writer 内部加锁

建议优先 **方案 B**，改动面更小，也更贴近责任边界。

## 澄清项 6：UI 节流的目标是“少刷”还是“聚合刷”？

### 问题

高频更新时，到底是简单丢弃部分更新，还是把更新聚合后再刷？

### 推荐答案

采用 **聚合优先、丢弃次之** 的策略。

### 推荐原因

1. 仅仅“少刷”容易丢掉关键状态，用户会觉得信息不完整。
2. 聚合后刷新可以同时满足性能和可读性，例如：
   - 当前动作显示“当前任务 + 当前文件 + 聚合分片进度”
   - 日志按批刷新，而不是每个 chunk 一行
   - 任务表只在状态真正变化时更新

### 具体建议

- `current_action`：按时间窗口聚合
- `append_log`：批量 flush
- `update_task_status`：仅状态变化时更新已有 item，不反复新建 item

## 澄清项 7：本轮是否要求补建测试体系？

### 问题

仓库当前没有现成 `tests/` 目录，本轮是否必须把测试体系一起建立？

### 推荐答案

**必须补建最小测试体系。**

### 推荐原因

1. 本次改动横跨 worker、executor、uploader、GUI 样式和节流逻辑，纯手测风险太高。
2. 并发改造最怕回归不可见；没有自动化测试，后续调默认值会非常危险。
3. 这里不要求一次建成完整集成测试平台，但至少要有单元测试 + GUI 基线测试 + benchmark smoke test。

## 建议直接确认的最终决策

若没有额外业务约束，建议把以下内容视为本轮默认决策：

1. 默认 `job_concurrency = 2`
2. 默认 `part_concurrency = 3`
3. 并发参数第一阶段只进配置，不上 UI
4. 登录弹窗第一阶段先统一状态与样式，不强制立即切 Qt 线程模型
5. benchmark 使用“小文件 / 大文件 / 混合”三套固定样本
6. result writer 默认按非线程安全处理，增加串行保护
7. UI 节流采用聚合刷新策略
8. 本轮必须补建最小测试体系

## 对实现顺序的影响

确认以上决策后，实施可直接按以下顺序推进，不再反复回头：

1. 测试骨架
2. 设置模型
3. job 并发
4. result writer / executor 并发兼容
5. multipart 并发
6. UI 样式统一
7. 登录弹窗状态统一
8. UI 节流
9. benchmark
10. 全量回归与文档
