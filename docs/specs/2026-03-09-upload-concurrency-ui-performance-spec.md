# 上传并发能力、UI 交互一致性与性能优化规格

## 1. 背景

当前项目是基于 Python 3.12 + PySide6 的 Windows 桌面上传工具。现有上传链路已经通过 `QThread` 避免主线程阻塞，但实际上传执行仍以串行为主，导致大批量文件或大文件任务的整体耗时偏长，无法满足用户对上传吞吐的预期。

同时，桌面 UI 的交互反馈覆盖不完整：部分按钮、标签页已定义 hover 或 checked 态，但输入框、树、表格、日志区、登录弹窗等控件缺少统一的 focus / hover / disabled / busy 反馈，导致交互感知不一致。随着上传任务变多，日志刷新、进度文本更新、表格状态变更也会进一步放大 UI 性能问题。

## 2. 现状调研结论

### 2.1 上传执行现状

- `src/quark_uploader/gui/workers.py`
  - `UploadWorker.run()` 对 `plan.jobs` 逐个顺序执行。
  - 当前并发仅体现在“把上传流程放进后台线程”，不等于多任务并发上传。
- `src/quark_uploader/services/upload_executor.py`
  - `UploadExecutionEngine.execute_job()` 对 `job.file_entries` 顺序上传。
  - 文件失败重试已存在，但没有文件级并发调度或并发限制。
- `src/quark_uploader/services/quark_file_uploader.py`
  - `_upload_multiple_parts()` 已支持 multipart，但 chunk 上传通过 `while offset < file_size` 顺序执行。
  - `build_upload_pre_payload()` 已携带 `parallel_upload` 语义，但本地代码尚未真正实现并发分片调度。

### 2.2 UI 与交互现状

- `src/quark_uploader/gui/main_window.py`
  - `_apply_styles()` 是主窗口 UI 样式与交互状态的集中定义点。
  - 已定义的状态主要覆盖按钮 hover、模式切换 checked、标签页 selected / hover、按钮 disabled。
  - `QLineEdit`、`QTreeWidget`、`QTableWidget`、`QPlainTextEdit` 缺少统一 focus 态规则。
- `src/quark_uploader/gui/official_login_dialog.py`
  - 登录弹窗独立构建界面，未复用主窗口统一样式体系。
  - Cookie 校验使用 `threading.Thread`，与主上传流程使用的 `QThread` 模型不一致。

### 2.3 性能现状

- 高频日志输出与进度文本更新可能造成 UI 重绘压力。
- 任务表格与日志区在大量任务下缺少更新节流策略。
- 当前缺少针对“并发上传提升吞吐，但不拖垮 UI”的性能边界定义。

## 3. 目标

### 3.1 功能目标

1. 将上传执行从“单线程串行上传”升级为“可配置的受控并发上传”。
2. 在不破坏现有上传结果正确性、分享生成、停止机制与错误回写能力的前提下，提高整体上传吞吐。
3. 建立统一的 UI 交互反馈规范，保证主窗口与登录弹窗的交互表现一致。
4. 降低高频 UI 更新带来的卡顿，保证在大批量任务下界面仍可响应。

### 3.2 非目标

1. 不改动夸克网盘接口协议本身。
2. 不引入新的第三方并发框架或 UI 框架。
3. 不重写整套界面结构，仅在现有 PySide6 架构上做增量优化。
4. 不承诺无限制并发；并发能力必须受配置和安全边界控制。

## 4. 设计原则

1. **受控并发优先**：并发数必须可配置，并设置安全默认值与上限，避免触发远端限流、内存放大或本地磁盘读取争抢。
2. **UI 主线程安全**：后台任务不得直接操作界面控件，所有 UI 变更必须通过 Qt signal/slot 归并回主线程。
3. **交互一致性优先于个别组件特例**：相同语义状态应有统一视觉反馈。
4. **性能优化以“减少无效更新”为核心**：优先节流、批量更新、降低冗余重绘，而不是单纯增加线程数。
5. **保持停止、重试、日志和结果写入语义不变**。

## 5. 方案概述

### 5.1 推荐方案

采用“两层受控并发 + 一层 UI 更新节流”的方案：

1. **任务级并发**
   - 在 `UploadWorker` 与上传执行层之间增加任务调度能力。
   - 支持同一 `UploadExecutionPlan` 中多个 `UploadJob` 并发执行。
   - 并发度可配置，例如默认 2，最大 4 或 6，由实现阶段结合稳定性验证确定。

2. **分片级并发**
   - 在 `QuarkFileUploader._upload_multiple_parts()` 中引入受控 chunk 池。
   - 仅对 multipart 大文件启用分片并发；小文件仍走单分片逻辑。
   - 分片并发必须保证最终 ETag 顺序、完成合并顺序和取消语义正确。

3. **UI 更新节流**
   - 对日志追加、当前动作文本、任务表格状态、进度条刷新进行频率控制与批量归并。
   - 减少每个 chunk 或每次细粒度状态变化都立即触发 UI 重绘。

4. **统一交互样式层**
   - 将主窗口可复用的颜色、边框、焦点、禁用、忙碌态规则抽象为统一样式契约。
   - 主窗口和 `OfficialLoginDialog` 统一接入这套契约。

### 5.2 备选方案与取舍

#### 方案 A：仅做任务级并发

- 优点：实现复杂度最低，风险相对可控。
- 缺点：大文件上传仍被单文件分片串行限制，吞吐提升有限。

#### 方案 B：仅做分片级并发

- 优点：大文件场景提升明显。
- 缺点：多文件批量场景收益不足，且 UI 层仍然感知为“单任务慢跑”。

#### 方案 C：任务级并发 + 分片级并发 + UI 节流（推荐）

- 优点：覆盖批量小文件与大文件两类核心场景，同时控制 UI 负载。
- 缺点：实现和测试复杂度最高，需要更严格的停止、重试与回归验证。

## 6. 功能需求

### FR-1 并发配置

系统必须提供上传并发配置能力，至少包括：

- 任务并发数（job concurrency）
- 分片并发数（part concurrency）
- 安全默认值
- 合理上限

配置可以来自本地设置或实现阶段约定的配置入口，但必须保证：

- 未显式配置时也有稳定默认值。
- 超过上限时自动回落到安全值。
- 非法值不会导致程序崩溃。

### FR-2 任务级并发上传

系统必须支持在一个上传计划内并发执行多个 `UploadJob`，并满足：

- 每个任务的状态、分享链接、重试次数、错误信息独立追踪。
- 停止操作发出后，不再启动新的待执行任务。
- 已运行中的任务遵循现有取消机制尽快停止。
- 结果输出结构与现有字段语义兼容。

### FR-3 分片级并发上传

对 multipart 文件，系统必须支持在安全并发限制下同时上传多个 chunk，并满足：

- 分片鉴权、上传、ETag 收集、complete 合并逻辑正确。
- 合并时使用正确的分片序号顺序。
- 任一分片失败时能够中止该文件上传，并按现有重试策略处理。
- 停止请求发出后，不再提交新的分片上传。

### FR-4 失败与重试兼容

现有文件级失败重试与分享重试语义必须保留，并满足：

- 重试只影响失败对象，不得污染其他任务状态。
- 并发场景下失败日志必须包含任务名/文件名/分片号等最小定位信息。
- 最终汇总结果必须准确反映成功、失败、停止数量。

### FR-5 统一停止语义

无论是任务级并发还是分片级并发，停止操作都必须满足：

- UI 立即反馈“正在停止”或等价状态。
- 不再调度新的 job / file / part。
- 允许已开始的网络请求在可取消点尽快结束。
- 停止后所有未执行项目状态保持一致，不出现部分悬空状态。

## 7. UX 与交互需求

### UX-1 统一交互状态

以下控件必须定义并体现统一的交互状态：

- `QPushButton`
- `QLineEdit`
- `QTreeWidget`
- `QTableWidget`
- `QPlainTextEdit`
- 登录弹窗中的主要按钮、状态文本和 WebView 容器

统一状态至少包括：

- 默认态
- hover 态
- focus 态
- disabled 态
- selected / checked 态（如适用）
- busy / uploading / validating 态（如适用）

### UX-2 上传过程可感知

用户必须能够清晰区分以下阶段：

- 等待上传
- 正在上传
- 正在重试
- 正在生成分享
- 正在停止
- 已完成 / 已失败 / 已停止

阶段变化必须在主窗口摘要区、任务表格和当前动作文本中至少一处得到稳定反馈。

### UX-3 登录弹窗一致性

`OfficialLoginDialog` 必须与主窗口共享统一的视觉语义，包括但不限于：

- 字体层级
- 按钮样式
- 状态文本颜色语义
- 焦点边框或高亮规则
- 忙碌/校验中状态提示

### UX-4 避免噪声式刷新

在高并发和高频分片上传场景下，界面不得因为每个细粒度状态变化而频繁闪烁或滚动跳动。日志区、当前动作区、进度区应优先展示对用户有意义的聚合信息。

## 8. 性能需求

### PF-1 吞吐提升目标

在相同网络与文件集条件下，并发上传方案应相较当前串行方案体现明确吞吐提升。实现验收时必须通过基准对比证明总体耗时下降。

> 注：具体提升比例可在实现前通过样本集校准；若未单独约定，至少应证明“默认并发配置下总体耗时优于当前串行实现”。

### PF-2 UI 响应性

上传期间主窗口必须保持可交互，至少包括：

- 标签页切换可用
- 停止按钮可点击
- 日志区可滚动
- 窗口拖动、最小化、恢复不出现明显卡死

### PF-3 UI 更新节流

系统必须对高频状态更新做节流或批处理，至少覆盖：

- 当前动作文本
- 日志追加
- 总进度刷新
- 表格状态刷新

### PF-4 资源边界

并发能力不得以无限放大 CPU、内存或句柄占用为代价。实现阶段必须定义并验证：

- 默认并发配置下的可接受资源占用
- 超大文件与大量小文件场景下不会出现明显失控增长

## 9. 可观测性与日志需求

1. 日志必须能区分任务级事件、文件级事件、分片级事件。
2. 并发场景下日志至少应携带以下上下文中的若干项：任务名、文件名、分片号、重试次数、阶段。
3. UI 展示应以聚合信息为主，详细诊断可保留在输出日志中。
4. 若远端返回限流或鉴权异常，日志必须能帮助判断是否与并发设置相关。

## 10. 兼容性与风险

### 10.1 兼容性要求

- 保持现有单文件、单文件夹、批量子文件夹三种入口不变。
- 保持现有结果写入、分享生成、停止按钮行为和主要输出字段兼容。
- 保持 Windows 桌面运行方式不变。

### 10.2 风险点

1. 远端接口可能存在隐藏限流或顺序依赖。
2. 分片并发会放大鉴权请求与网络错误暴露概率。
3. 并发日志与 UI 刷新若不节流，可能抵消吞吐收益。
4. 登录弹窗与主窗口若继续使用不同线程模型，后续维护成本会继续升高。

### 10.3 风险应对

1. 通过受控并发上限与默认值降低远端压力。
2. 为任务级与分片级错误补充最小必要上下文。
3. 对 UI 更新做统一节流策略。
4. 优先统一后台执行模型与样式契约。

## 11. 验收标准

### AC-1 并发能力验收

- 在默认配置下，同一上传计划中的多个 job 可以并发执行。
- 大文件 multipart 上传可在受控并发下同时处理多个分片。
- 停止后不再启动新的 job 或 chunk。

### AC-2 正确性验收

- 上传完成后，远端文件/目录结果与串行版本一致。
- 分享链接生成逻辑在并发场景下仍正确。
- 失败、重试、停止状态不会相互串扰。

### AC-3 UI 一致性验收

- 主窗口与登录弹窗的核心控件具备统一的 hover / focus / disabled / busy 反馈。
- 用户在鼠标悬停、键盘聚焦、上传中、停止中等状态下看到的视觉语义一致。

### AC-4 性能验收

- 默认并发配置下，总体上传耗时优于当前串行版本。
- 上传过程中 UI 仍可响应主要交互。
- 在大量日志与状态更新下，不出现明显卡顿、闪烁或假死。

### AC-5 回归验收

- 单文件上传回归通过。
- 单文件夹上传回归通过。
- 批量子文件夹上传回归通过。
- 停止上传回归通过。
- 登录弹窗获取 Cookie 流程回归通过。

## 12. 建议实施拆分

### Phase 1：并发基础设施

- 引入上传并发配置模型。
- 建立任务级并发调度骨架。
- 保证取消、状态回传与结果聚合正确。

### Phase 2：分片并发与错误控制

- 为 multipart 引入分片并发池。
- 补齐分片失败、重试、ETag 顺序与 complete 合并验证。

### Phase 3：UI 一致性与性能优化

- 抽离统一样式契约。
- 主窗口与登录弹窗接入统一交互状态。
- 为日志与进度刷新增加节流/批量更新机制。

### Phase 4：验证与调优

- 建立串行 vs 并发基线对比。
- 调整默认并发值与上限。
- 补齐回归测试与手动验证清单。

## 13. 打开问题

1. 默认任务并发数与分片并发数应设为多少，才能兼顾网络收益与稳定性？
2. 是否需要在 UI 中暴露并发设置，还是仅保留在本地配置中？
3. 登录弹窗是否需要统一切换到 Qt 线程模型，还是只统一样式与状态反馈？
4. 性能验收采用哪些固定样本文件集，才能得到稳定可复现的对比结果？
